Java 必知必会——分层思维

发表于 2021-07-25 分类于 Java

1. 软件架构

软件程序通常由两方面需求组成：

  • 功能性需求：程序能为用户做什么，比如，查看订单信息、下单等。
  • 非功能性需求：除功能性需求以外的其他必要需求，比如，访问性能、安全性、容错和恢复、本地化、国际化等。

非功能性需求所构建起来的正是我们所认知的软件架构。

什么是软件架构？

简单来说就是软件组成的基本结构，主要为三个要素：代码、代码之间的关系和两者各自的属性。

最经典的软件架构就是分层架构，也就是将软件系统分层。

2. 什么是代码分层架构？

想知道代码分层结构是什么，就要从软件部署分层架构开始。

常见的软件分层架构，主要有四个部分组成：

  • 客户端（Client）：也就是调用方，比如浏览器和 APP。
  • Web 服务器（Web Server）：实现程序的运行逻辑，并从缓存或数据层获取数据，返回给上层客户端。
  • 应用服务器的缓存（Cache）：加速存储数据的访问。
  • 数据层（Database）：存储数据。

简单来说，软件分层架构就是通过层来隔离不同的关注点（变化相似的部分），以此来解决不同需求变化的问题，让这种变化可以被控制同一个层中。

软件开发者通常更关心的是应用程序（Web Server）的分层架构，比如常见的 MVC 分层架构。其他的如 Database、Cache等都有自己的代码分层架构。

那么，什么是代码分层架构？

代码分层架构就是将软件 “ 元素 ”（代码）按照 “ 层 ” （代码关系）的方式组织起来的一种结构。

分层架构的核心原则：当请求或数据从外部传递过来后，必须从上一层传递到下一层。

3. 代码分层架构解决的问题

代码分层架构主要解决两个问题：

  • 如何快速拆解功能问题？
  • 如何提升代码的可扩展性？
3.1 通过分层来拆解问题

软件开发过程中，一个功能需求问题一般都是笼统的复杂问题，需要将这个笼统的复杂问题拆分为多个层次的子问题来解决。

比如用户短信登录验证这个功能？可以拆解为：

  1. 验证用户是否存在
  2. 发送短信验证码
  3. 校验短信验证码

这是最初问题的第一次拆分，在具体到每个细节点：

  1. 验证用户是否存在
    • 获取用户输入内容（View+Controller）
    • 查询用户是否存在（Model）
    • 返回结果（View）
  2. 发送短信验证码
    • 获取用户手机号（View+Controller）
    • Api 向手机发送登录验证码 (Model)
    • 60s 之后才能再次发送，避免用户频繁触发验证码发送（View）
  3. 校验短信验证码
    • 获取用户输入验证码（View+Controller）
    • Api 向验证登录验证码 (Model)
    • 返回结果（View）

不知不觉中，通过分层就把一个复杂的大问题拆解为了多个容易解决的子层问题，可能很多子层问题已经被解决了，比如发送短信验证码、校验验证码等。最后，真正需要关注的问题实际上变少了。

因此，从功能性需求触发，代码分层本身就是一种拆解复杂问题的好方法。

3.2 通过分层来提升代码可扩展性

为什么要提高代码可扩展性？

因为真实的系统数据一直在不断增长，比如存储的数据从一百万增长到一千万、一个亿，单体架构往往难以承载这种数据量，往往是因为扩展服务器和数据库功能时，一个的代码修改就可能影响到整个系统功能。

分层架构可以将复杂的逻辑切分为多层，将大问题变成了小问题。

这样每个小问题就容易解决，容易被抽象为一个组件，当组件功能需要扩充或者替换时，修改代码的影响也被有效控制在有限范围内，此时，组件自身的复用性也提高了。

除此之外，分层架构还更容易做服务的横向扩展。

简单来说，可以用多台服务器共同提供服务，也就是集群部署。比如将 Model 层单独抽取出来作为数据服务层部署，这样不影响其他业务层，同时负载增大时，仅扩展 Model 层即可扩展整个系统的承载能力。

4. 代码分层架构优缺点

4.1 代码分层架构优点
  • 只需关注整体结构其中某一层的具体实现。
  • 降低了层与层之间的依赖。
  • 容易使用新的实现来替换原有层的实现。
  • 有利于标准化统一。
  • 各层逻辑便于复用。

总的来说，代码分层架构设计主要是为了实现责任分离、解耦、组件复用和标准指定。

由于层本身就是一种组件形式，层与层划分后，可以通过统一的接口与外界进行交互，而不是按照功能上的依赖进行交互。

这里，统一的接口是模块之间互相约定的统一标准，只要按照标准来进行代码实现，就不会因为代码改动而影响到接口的使用。

4.2 代码分层架构缺点
  • 开发成本变高：因为各层责任不同，如果高层添加功能，需要各个低层来提供支持，难免会增加开发成本。
  • 性能降低：请求数据经过多层代码处理，执行时间变长，性能会有所消耗。
  • 代码复杂度增加：因为层与层之间存在强耦合，因此一些组合功能的调用，需要增加很多层之间的相互调用。
